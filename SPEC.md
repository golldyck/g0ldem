## 1. Видение и в чём суть (Why)

Сегодня хитовые крипто-AI-агенты (Luna, aixbt, Truth Terminal на Virtuals) — это **спекулятивные токены поверх чёрных ящиков**. Покупая токен агента, ты:
- не владеешь самим интеллектом (он централизован у команды),
- не можешь доказать, что отвечает ИИ, а не человек,
- не можешь перенести/запустить «мозг» сам.

**G0LDEM закрывает это.** Ты вайбкодишь агента словами (вкладываешь «шем») → он получает зашифрованный «мозг» (персона + навыки + растущая память) → минтится («forge») как **INFT (ERC-7857)** на 0G → его можно **продать** (интеллект физически переезжает к покупателю через ре-шифрование) или **сдать в аренду** (AI-as-a-Service без передачи владения).

Главный дифференциатор и для судей, и для голосования сообщества:
**«True AI Ownership vs API Access»** — ты владеешь самим интеллектом, он верифицируем (0G Compute), портативен (переезжает при продаже), приватен (шифрование в 0G Storage).

**Killer-демо (30 сек):** «Описываю агента фразой → он отвечает + пруф что это ИИ → минчу как INFT → продаю на другой кошелёк → теперь Я НЕ МОГУ им пользоваться, а покупатель может. Мозг переехал.» Virtuals этого показать не может физически.

## 2. Что агент умеет (Functions)

Агент = **persona** (характер) + **skills** (инструменты, которые он может вызывать) + **memory** (растущее зашифрованное состояние) + **verifiable activity log** (доказуемая история).

### Типы агентов (платформа ниша-агностик, но в демо показываем ОДИН)
| Тип | Что делает | Демо-сила |
|---|---|---|
| Companion / NPC | диалог, роль, помнит отношения с юзером | 🔥 высокая (эмоция = голоса) |
| Analyst | анализ рынка/данных, верифицируемые «коллы» | средняя (нужен контекст) |
| Trader / DeFAI | исполняет стратегию на тестнете, верифицируемый PnL | средняя (доверие долго строится) |
| Creator | генерит контент/мемы/арт в выученном стиле | высокая (шеринг) |
| Assistant | автоматизация задач, персонализирован под владельца | средняя |

**Рекомендация для турнира:** killer-агент для демо — **Companion/NPC или Creator (роаст-бот / мем-генератор)**. Эмоция и шеринг собирают голоса лучше, чем трейдинг.

### Базовые операции агента
- `chat(input)` — инференс через 0G Compute с верификацией (возвращает ответ + proof).
- `learn(interaction)` — дописывает память в зашифрованное состояние на 0G Storage (повышает ценность).
- `useSkill(skillId, args)` — вызов навыка/инструмента (например, выложить пост, посчитать метрику).
- `exportProvenance()` — отдаёт верифицируемую историю активности для оценки ценности.

## 2.1 На чём построен агент: база, мозг, модель

### Из чего состоит агент
Агент — это не одна модель, а **«мозг» = конфиг + память + лог**, который интерпретируется LLM на 0G Compute:
- **Persona / character file** — формат в духе ElizaOS (де-факто стандарт крипто-агентов): `name`, `bio`, `lore`, `style`, `topics`, `adjectives`, `messageExamples`. Это «характер».
- **Skills** — список инструментов/действий, которые агент может вызывать (`useSkill`): постинг, расчёт метрики, вызов контракта и т.п.
- **Memory** — растущее состояние (краткосрочное + долгосрочное), которое дописывается при `learn()`. Чем больше — тем «прокачаннее» и дороже агент.
- **Activity log** — верифицируемая история (proof'ы из Compute, доход on-chain) для расчёта AVS.

### Где «мозг» (где хранится)
- Весь мозг (persona + skills config + memory) **шифруется (AES-256-GCM) и кладётся в 0G Storage**.
- В INFT хранится только `encryptedURI` + `metadataHash` (ссылка + хеш), сам интеллект — приватен.
- При продаже мозг ре-шифруется оракулом под покупателя (см. раздел 4.2).

### Какая AI отвечает (inference)
Инференс идёт через **0G Compute (Private Computer)** — это **OpenAI-совместимый API**, где каждый запрос исполняется в TEE-enclave и ответ подписывается приватным ключом TEE (это и есть наш «verified AI» proof). Оплата — в токене 0G.

Реально доступные на 0G Compute модели (сверить актуальный список перед стартом):
- **GLM-5-FP8** — *рекомендуемая по умолчанию*: reasoning + thinking mode, **tool-calling из коробки**, API совместим и с OpenAI, и с Anthropic, FP8 = быстро. Идеально для агента, который вызывает скиллы.
- **DeepSeek chat-v3** — мощный reasoning (685B MoE), дёшево.
- **Qwen3.6 Plus** — контекст до **1M токенов** → отлично для агентов с большой растущей памятью.
- **Qwen3-VL-30B** — мультимодальный (текст+картинки) для visual-агентов/компаньонов.
- **Whisper-large-v3** (речь), **z-image** (генерация картинок) — для creator/voice-агентов.

Доступ: **Router** (`router-api.0g.ai/v1`) сам выбирает провайдера и тянет оплату из единого баланса; **Advanced mode** — если нужен контроль конкретного провайдера. Миграция с OpenAI — «меняется только конструктор», остальное (chat completions, streaming, tool-calling) совместимо.

### Verification modes (важно для «доказуемости»)
- **TeeML** — модель крутится прямо внутри TEE, ответ подписан ключом enclave (для self-hosted: GLM-5-FP8, DeepSeek, GPT-OSS-120B). Это наш основной режим для proof.
- **TeeTLS** — брокер в TEE проксирует запрос к внешнему провайдеру по HTTPS с криптопруфом аутентичного роутинга.

## 3. Как оценивается ценность агента (Valuation Engine) — ЯДРО ПРОЕКТА

Это главная фича, которой нет у конкурентов. Идея: **каждый вход в формулу ценности криптографически верифицируем** (proof из Compute, on-chain доход, хеш-коммиты памяти в Storage), поэтому track record нельзя подделать — в отличие от Virtuals, где «история» это просто заявление.

### Agent Value Score (AVS) — компоненты
1. **Training Depth (глубина прокачки).** Объём/богатство накопленной памяти + число верифицированных взаимодействий. Источник: хеш-коммиты состояния в 0G Storage + счётчик инференсов с proof из Compute.
2. **Verified Track Record (доказанный результат).** Для трейдера — PnL с пруфами; для аналитика — success rate коллов; для creator/companion — usage count + удержание. Ключ: метрики подтверждены proof'ами, а не вписаны руками.
3. **Reputation (репутация).** Рейтинги от тех, кто арендовал агента (AIaaS), взвешенные по объёму верифицированного использования (нельзя накрутить отзывами без реального юзинга).
4. **Revenue History (денежный поток).** Накопленный on-chain доход от аренды/использования. Агент, заработавший X, — это актив с доказуемым доходом (как cashflow-актив).
5. **Capability & Rarity (навыки и редкость).** Набор скиллов, уникальность персоны, редкие способности.

### Формула (стартовая, настраивается)
```
AVS = w1*log(1 + verified_interactions)
    + w2*normalized_track_record        // 0..1, доказан proof'ами
    + w3*reputation_score               // 0..1, взвешен по верифиц. юзингу
    + w4*log(1 + cumulative_revenue)
    + w5*capability_score               // ширина/редкость скиллов
```
- Веса `w1..w5` — конфиг контракта/бэка, тюнятся.
- **Всё, что входит в AVS, должно иметь источник-пруф.** Если метрику нельзя верифицировать — она не влияет на цену. Это и есть нарратив 0G.
- AVS считается off-chain (бэк), но **инпуты коммитятся on-chain/в Storage с хешами**, чтобы любой мог пересчитать и проверить. Опционально — on-chain view-функция `getValuationInputs(tokenId)`.

### Definition of Done для Valuation (MVP)
- Контракт/Storage хранит верифицируемые счётчики: `verifiedInteractions`, `cumulativeRevenue`, `reputationCommits`.
- Бэк считает AVS и показывает разбивку по компонентам на карточке агента.
- На UI видно: «эта цифра подтверждена пруфом X» (ссылка на proof/commit).

## 4. Безопасность (Security) — обязательный раздел

INFT хранит зашифрованный интеллект; вся ценность держится на том, что приватность и переходы безопасны. Модель угроз и меры:

### 4.1 Конфиденциальность «мозга»
- Мозг **всегда зашифрован** (AES-256-GCM), plaintext никогда не пишется в Storage/логи.
- Ключ шифрования держит владелец (sealed key под его публичный ключ).
- Случайный IV/salt на каждое шифрование; KDF (pbkdf2/scrypt) если ключ из пароля.

### 4.2 Безопасный перенос (re-encryption на продаже)
- Оракул (TEE / в перспективе ZKP) ре-шифрует метаданные под ключ покупателя; старый ключ продавца инвалидируется.
- Контракт в `transfer(from, to, tokenId, sealedKey, proof)` **проверяет proof** через `IOracle.verifyProof`.
- **Пруфы имеют срок жизни** → включить nonce + expiry, защита от replay.
- Для демо: TEE-путь из референс-репо. Для прода: децентрализованный набор оракулов + TEE-аттестация.

### 4.3 Управление ключами
- Приватные ключи только в кошельке/HSM, никогда в коде/логах/`.env` в репо.
- Подпись операций через кошелёк (wallet signing), не хранить ключи на бэке.
- Ротация ключей; для критичных операций — multisig.

### 4.4 Смарт-контракт
- `ReentrancyGuard` на `transfer`/маркетплейс-операциях.
- Access control: `ownerOf`/`onlyOwner` проверки на mint/authorize/list.
- Checks-Effects-Interactions; валидация всех внешних входов (proof, sealedKey длины).
- **Атомарность сделки:** оплата и ре-шифрование/трансфер — через escrow, чтобы нельзя было заплатить и не получить мозг (или наоборот).
- Защита от front-running на покупке (escrow / commit-reveal / привязка к покупателю).
- **Аудит перед mainnet** (на хакатоне — тестнет, но в спеке отметить).

### 4.5 Инференс и целостность
- Верифицируемый Compute (TEE/ZKP) — результат инференса нельзя подделать. От этого зависит честность Valuation.
- Любой proof, влияющий на AVS, проверяется до записи метрики.

### 4.6 Память и data poisoning
- Злонамеренные взаимодействия могут «отравить» память агента.
- Меры: владелец контролирует, что коммитится в долгую память; коммиты памяти подписаны и хешируются (провенанс); опц. модерация перед `learn()`.

### 4.7 Аренда (authorizeUsage)
- Скоупленные права: `maxRequests`, `allowedOperations`, `rateLimit`, `expiresAt`.
- Возможность отзыва (revoke) досрочно.
- Арендатор НЕ получает ключ к мозгу — только право на инференс через Compute под контролем контракта.

### 4.8 Приватность владельца
- Для personal-assistant агентов — чувствительные данные владельца шифруются и контролируются им; «право на удаление».

## 5.1 Как выглядит INFT (визуальный дизайн)

Мозг агента зашифрован, поэтому публичное «лицо» INFT — это **карточка-паспорт агента**: аватар + верифицируемые статы + тир. Ключевая идея: **карточка визуально отражает ценность** — чем больше агент прокачан и заработал, тем выше его тир и тем «дороже» он выглядит. Это и есть dynamic NFT.

### Элементы карточки
- **Аватар-сигил** — генерируется **детерминированно из сида = hash(persona)**. Один характер → один уникальный внешний вид. Рендер: SVG/canvas по параметрам, сгенерированным из хеша (форма, палитра, символы).
- **Имя + тип** агента (Companion / Analyst / Trader / Creator / Assistant).
- **AVS** (Agent Value Score) — крупным числом.
- **Tier-бейдж** — визуальный ранг: Bronze → Silver → Gold → Prismatic, привязан к порогам AVS.
- **Training depth / level** — прогресс-бар прокачки.
- **Верифицируемые статы** — interactions, track record, reputation, revenue; у каждого значок «verified» со ссылкой на proof/commit.
- **Skills** — теги навыков.
- **Footer** — «brain on 0G Storage · <модель>» + адрес владельца + token id.

### Эволюция внешнего вида (dynamic NFT)
По мере роста AVS/level карточка меняется → создаёт визуальный стимул прокачивать и коллекционировать:
- рамка/тир меняет цвет (бронза → золото → призма);
- появляются бейджи за верифицированные достижения (например, «1k verified chats», «profitable 30d»);
- интенсивность ауры/деталей сигила растёт с уровнем;
- «прокачанный» агент явно отличается от «свежего».

### Как это реализовано технически
- `tokenURI` указывает на **публичные** метаданные карточки (имя, тип, тир, аватар-параметры, ссылки на proof'ы AVS) — **без раскрытия мозга** (мозг отдельно, зашифрован, в `encryptedURI`).
- Карточка ре-рендерится при изменении метрик (dynamic). Можно генерить SVG on-the-fly из текущего AVS/level.
- Стиль: флэт, тёмная тема в духе бренда 0G (zero-gravity / неон-пиксель-койн), без перегруза.

> Референс внешнего вида карточки — мокап, показанный в чате (Nyx-7 · Roast NPC). Перенести этот макет в `apps/web/agent/[id]`.

## 5.2 Картинка NFT при минте (image generation)

При минте у токена сразу должна быть картинка. Она **двухслойная**: неизменное «лицо» (Soul Sigil) + динамическая рамка тира.

### Слой 1 — Soul Sigil (identity, детерминированный) — ДЕФОЛТ под MVP
- `seed = keccak256(canonical_persona_json)` (name + bio + type + createdAt — только публичные поля).
- Детерминированный PRNG из `seed` выводит параметры узора: tier-палитра, число нод (symmetry order), число колец, форма ядра (glyph).
- Рендер в SVG. Те же черты → тот же sigil (как отпечаток).
- Плюсы: мгновенно, бесплатно (без инференса), воспроизводимо, **верифицируемо** (любой воспроизводит sigil из `persona_hash` и проверит соответствие), можно положить **полностью on-chain** как `data:image/svg+xml`.

### Слой 2 — Tier Frame (presentation, динамический)
- Рамка/аура/бейджи пересобираются из текущего `AVS`/`level` (Bronze → Silver → Gold → Prismatic).
- Реализация: dynamic `tokenURI`, который рендерит SVG с текущими статами (on-chain рендер ИЛИ off-chain рендерер-эндпоинт + снапшот в 0G Storage).

### Опция B — AI-портрет («вау», для companion/creator)
- Генерация через модель **z-image на 0G Compute** из *публичного* summary персоны → загрузка в 0G Storage → URI становится `image`.
- Минусы: один инференс (стоимость), недетерминирован, нужно пинить в Storage. Sigil остаётся как fallback/водяной знак.
- Гибрид: AI-портрет = лицо, sigil + тир = рамка.

### Где хранится
- Sigil: on-chain (`data:` URI) ИЛИ снапшот в 0G Storage.
- AI-портрет и полная карточка: **0G Storage** (content-addressed, по хешу).
- В контракте — `tokenURI` (публичные метаданные) и отдельно `encryptedURI` (зашифрованный мозг). **Картинка ≠ мозг.**

### Static vs dynamic
- Sigil — **immutable** birth-identity (привязан к `persona_hash`), не меняется → маркетплейсы стабильны.
- Frame/tier/badges — **dynamic**, отражают текущее состояние агента.

### Mint flow (по шагам)
1. Финализировать персону (vibe coding) → посчитать `persona_hash`.
2. Сгенерировать Soul Sigil из `persona_hash` (instant).
3. *(опц.)* Сгенерировать AI-портрет через z-image → залить в 0G Storage → `imageURI`.
4. Собрать **публичные** метаданные JSON (image + attributes) → залить в 0G Storage → `tokenURI`.
5. Зашифровать мозг → залить в 0G Storage → `encryptedURI`.
6. Заминтить INFT с `(tokenURI, encryptedURI, metadataHash)`.
7. При прокачке — dynamic `tokenURI` пересобирает карточку с текущим тиром/level.

### Публичные метаданные (форма)
```json
{
  "name": "Nyx-7",
  "description": "Roast NPC companion",
  "image": "<0G Storage URI или data:image/svg+xml>",
  "attributes": [
    {"trait_type": "Type", "value": "Companion"},
    {"trait_type": "Tier", "value": "Gold"},
    {"trait_type": "Level", "value": 6},
    {"trait_type": "AVS", "value": 847}
  ],
  "persona_hash": "0x…"
}
```

### Безопасность картинки
- Картинка генерится **только из публичных полей персоны** — зашифрованный мозг в неё не попадает никогда.
- `persona_hash` в метаданных позволяет любому перепроверить, что sigil соответствует агенту (анти-подмена образа).

## 6. Tech Stack (решения зафиксированы)

- **Контракты:** Solidity ^0.8.19, Hardhat, OpenZeppelin (ERC721, Ownable, ReentrancyGuard).
- **SDK 0G:** `@0gfoundation/0g-ts-sdk` (Storage), 0G Compute SDK (inference), `ethers` v6.
- **Бэкенд:** Node.js + TypeScript (API: компиляция персоны, инференс-прокси, расчёт AVS, подготовка transfer).
- **Фронт:** Next.js + React + TypeScript, `wagmi` + `viem` для кошелька/контрактов, Tailwind.
- **Ускорители:** 0G Agent Skills для Cursor (14 скиллов) + 0G MCP-серверы под операции с чейном/стораджем (см. Resources) — использовать для скорости.
- **Дизайн:** **Claude Design** (canvas + дизайн-инструменты, итерации через чат) — для всей визуальной части; реализация — в Claude Code.

### Дизайн-воркфлоу (Claude Design ↔ Claude Code)
Разделение труда, чтобы не путать «как выглядит» и «как работает»:

- **Claude Design делает визуал** (итерируем в чате на канвасе, экспортируем макеты):
  - дизайн карточки-паспорта агента (раздел 5.1) — финальные стили, тёмная тема в духе 0G;
  - эстетика **Soul Sigil** и рамок тиров (Bronze → Silver → Gold → Prismatic) — задаёт визуальный язык, который потом воспроизводит код-генератор;
  - экраны приложения: `train` (вайбкодинг-форма), `agent/[id]` (карточка), `market`;
  - шеринг-визуалы и питч/one-pager под раунды народного голосования (четвертьфинал+).
- **Claude Code делает реализацию:**
  - детерминированный генератор сигила из `persona_hash` (по визуальному языку из Claude Design);
  - контракты, SDK-обёртки, фронт-вёрстка по макетам, dynamic `tokenURI`.

> Поток: макет/стиль в **Claude Design** → передаём дизайн-токены и SVG-референсы в репозиторий → **Claude Code** реализует и подключает к данным. Sigil-генератор повторяет визуальный язык, заданный в Claude Design, но управляется хешем.

## 7. Build-план по фазам (мапится на этапы турнира)

> Правило: не переходить к следующей фазе, пока не выполнен **Definition of Done (DoD)** текущей.

### Phase 0 — Setup (день 1)
- Инициализировать монорепо по структуре выше. Hardhat init, Next.js init, TS везде.
- Подключить кошелёк к Galileo testnet, получить токены из фасета.
- Установить `@0gfoundation/0g-ts-sdk ethers @openzeppelin/contracts hardhat`.
- Поднять «hello world»: задеплоить пустой ERC721 на тестнет, прочитать из фронта.
- **DoD:** контракт задеплоен на тестнет, фронт читает его, кошелёк подключается.

### Phase 1 — Train + Mint + Verifiable Chat (→ Top 32, судят судьи)
- `metadata.ts`: собрать «мозг» (persona JSON: bio/lore/character/skills + init memory) → зашифровать → залить в 0G Storage → вернуть `{encryptedURI, sealedKey, metadataHash}`.
- Вайбкодинг-форма: промпт → компиляция персоны через 0G Compute → preview-чат с агентом.
- `inference.ts`: инференс через Compute с **verification mode**, возвращать ответ + proof; на UI показать бейдж «verified AI» со ссылкой на proof.
- **Image:** сгенерировать Soul Sigil из `persona_hash` (детерминированный SVG) → собрать публичные метаданные (image + attributes) → `tokenURI`. См. раздел 5.2.
- `INFT.sol`: `mint(to, tokenURI, encryptedURI, metadataHash)`; задеплоить.
- Карточка агента `/agent/[id]` (рендер из tokenURI + sigil).
- **DoD:** юзер описывает агента → чатится с ним (с proof) → минтит INFT → видит его на карточке. Это уже сабмит на Top 32.

### Phase 2 — Secure Transfer (коронный момент демо)
- `transfer.ts`: `prepareTransfer` (оракул ре-шифрует под покупателя) → `executeTransfer`.
- `INFT.sol`: `transfer(from, to, tokenId, sealedKey, proof)` с `verifyProof`, nonce+expiry, ReentrancyGuard.
- `MockOracle.sol` (TEE-путь) для тестнета.
- UI: кнопка «Sell to wallet» + демонстрация, что продавец теряет доступ, покупатель получает.
- **DoD:** мозг переезжает между двумя кошельками; продавец больше не может инференсить, покупатель может. Демо-флоу записывается.

### Phase 3 — Valuation + Marketplace (→ начинается голосование, нужен шеринг)
- `valuation.ts`: считать AVS из верифицируемых инпутов; коммитить инпуты (хеши) в Storage/on-chain.
- `learn()`: память растёт от использования → видимый «уровень прокачки» на карточке.
- `Marketplace.sol`: листинг + escrow + покупка (атомарно оплата↔трансфер).
- Публичная шеринг-карточка агента (для голосов) + лидерборд по AVS.
- **DoD:** агент имеет видимую, объяснённую пруфами цену; его можно листить и купить; карточкой можно поделиться.

### Phase 4 — Rental (AIaaS) + Polish (→ финал, делаем вирусным)
- `authorizeUsage(tokenId, executor, permissions)`: аренда без передачи владения; rate limit, expiry, revoke.
- «Попробуй чужого агента, не покупая» — демо аренды.
- Killer-нишевый агент для демо (NPC/роаст-бот), отполировать UX и 30-сек демо-сценарий.
- **DoD:** агента можно арендовать; есть отполированный демо-сценарий под голосование.
